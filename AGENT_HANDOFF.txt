THE ONE THING I WOULD WARN YOU ABOUT:

The widget system (Session 17) and the existing render functions are currently
in a hybrid state. The widgets mostly delegate to the old render functions
(renderJobs(), renderFort(), etc.) rather than reimplementing them. This is
intentional — it meant the migration could happen without rewriting 2000 lines
of render logic at once. But it means there are two parallel layers when there
should eventually be one. If you refactor, be careful about which layer owns
what, and test thoroughly — the element ID contracts between the render functions
and gameTick are load-bearing and not immediately obvious.

  WHAT THIS PROJECT IS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

An incremental RPG running entirely in the browser (HTML + vanilla JS, no
framework). The player starts as a lone mercenary commander in an abandoned
fort and builds a private military company that eventually becomes a sovereign
nation-state, powerful enough that actual nations fear and actively try to
suppress it. The thematic reference is Outer Heaven / Diamond Dogs from the
Metal Gear Solid series — the endgame goal is PMC Tier 6: "Sovereign Force".

The game must be served over HTTP (not opened as a file:// URL) because it uses
fetch() to load JSON data. Use server_manager.pyw to do this — it is a
Python/tkinter GUI that starts/stops python -m http.server, monitors activity,
and can export the log. Double-click it or run: pythonw server_manager.pyw

  THE CARDINAL RULE — DATA DRIVEN, NO HARDCODED IDs
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The most important architectural decision in this project:

  ** ADDING CONTENT SHOULD NEVER REQUIRE EDITING A JS FILE **

All game content — contracts, recruit classes, upgrades, nations, milestones,
tiers, suppression events — lives in JSON files in /data/. The JS is a generic
interpreter, not a list of hardcoded cases.

This was achieved deliberately and took several refactor sessions. Do not break
it. If you find yourself writing "if (upgrade.id === 'something')" in a JS file,
stop and find the data-driven way instead.

The mechanism is the effects[] array in fort_upgrades.json and the bonuses[]
array in recruit_classes.json. These are read by js/engine.js which accumulates
them generically. See the EFFECT TYPE REFERENCE section below.

  THE ENGINE (js/engine.js) — HOW IT WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The engine never names an upgrade ID or class name. It works through two
accumulators:

  getActiveUpgradeEffects()
    → Scans all built upgrades, collects their effects[] arrays.
    → Returns: { effectType: [{value, categoryFilter?}, ...], ... }

  getActiveRosterEffects()
    → Same pattern but for roster member bonuses[].

  accumulateMult(effects, category)
    → Multiplies (1 + value) for every matching effect.
    → An effect with categoryFilter only applies when category matches.

  accumulateAdd(effects, category)
    → Sums value for every matching effect.

Every derived quantity in the game goes through this:

  getJobDuration(job)   → durationMult effects from upgrades + roster
  getJobReward(job)     → rewardMult effects from upgrades + roster
  getFameReward(job)    → fameMult effects + config dangerFameDefaults
  getJobXp(job)         → xpMult effects
  getHpRegenRate()      → hpRegen + hpRegenMult + hpRegenBonus
  maxRoster()           → baseRosterSize (config) + rosterSlots effects
  getCommanderStatBonus() → commanderStatBonus effects
  getMagicBonus()       → origin magic + magicBonus effects × magicMultiplier
  companyTotalMagic()   → roster magic sum + getMagicBonus()
  getNationThreatMult() → 1.0 + nationThreatMult effects (negative = less threat)
  getSuppressionResist() → suppressionResist effects sum
  getActiveJobCap()     → base 1 + activeJobCap effects

To add a new mechanic, add a new effect type to the JSON, then add a function
to engine.js that calls accumulateAdd or accumulateMult on that type. Done.

  EFFECT TYPE REFERENCE (fort_upgrades.json effects[] and recruit bonuses[])
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All current effect types and what they do. All support optional "categoryFilter"
to restrict to a specific job category (e.g. "arcane", "combat", "intelligence").

  rewardMult          Multiplies contract reward. value: 0.10 = +10%.
  durationMult        Multiplies contract duration. value: -0.15 = -15%.
  fameMult            Multiplies fame earned per contract.
  xpMult              Multiplies XP earned per contract.
  xpReqMult           Multiplies XP required to level up. Use negative values.
  hpRegen             Base HP regen rate per day (fraction of maxHp).
  hpRegenMult         Additive bonus to base hpRegen rate.
  hpRegenBonus        Same as hpRegenMult (recruit bonus variant).
  rosterSlots         Additive bonus to max roster size.
  commanderStatBonus  Additive bonus applied to all commander stats for checks.
  magicBonus          Additive bonus to company-wide magic total.
  magicMultiplier     Multiplies ALL magic bonuses (including origin magic).
  nationThreatMult    Adjusts threat accumulation multiplier. value: -0.25 = 25%
                      less threat generated from contracts.
  suppressionResist   Fraction reduction in suppression event effectiveness.
  activeJobCap        Additive bonus to maximum simultaneous active contracts.
  blockadeImmunity    Boolean. If true, economic blockade penalties are ignored.
  fortWarded          Boolean. Marks the fort as warded (unlocks void contracts).
  unlockRareRecruits  Boolean. Allows rare-rarity classes into the recruit pool.
  recruitStatBonus    Additive stat bonus applied when generating all recruits.
                      Requires "stat" field: "str", "end", "lead", or "cun".

To add a new effect type: add it to JSON, add a function in engine.js that reads
it, call that function wherever the effect should apply.

  MILESTONE CHECK TYPE REFERENCE (milestones.json check.type)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Evaluated in js/logic/progression.js → evaluateMilestoneCheck(check).

  totalJobsDone         GS.totalJobsDone >= check.value
  rosterLength          GS.roster.length >= check.value
  rosterFull            GS.roster.length >= maxRoster()
  bronze                GS.bronze >= check.value
  fame                  GS.fame >= check.value
  commanderLevel        GS.commander.level >= check.value
  upgradeCount          Built upgrade count >= check.value
  allUpgrades           All upgrades built
  upgradeBuilt          GS.upgrades[check.value] is truthy
  fortWarded            isFortWarded() (engine function)
  rosterHasClass        Any roster member has cls === check.value
  rosterHasRarity       Any roster member has rarity === check.value
  hasRareRecruit        Any roster member has rarity === 'rare'
  hasMagicUser          Any roster member has magic >= magicUserThreshold
  totalMagicStat        companyTotalMagic() >= check.value
  completedJobCategory  GS.completedCategories[check.value] > 0
  categoryJobsDone      GS.completedCategories[check.category] >= check.value
  categoryJobsDoneMulti Sum of completions across check.categories >= check.value
  totalStr / totalCun / totalEnd  Company total stat >= check.value
  specificJobDone       GS.completedJobs[check.value] > 0
  specificJobDoneCount  GS.completedJobs[check.job] >= check.value
  pmcTier               GS.pmcTier >= check.value
  suppressionSurvived   GS.suppressionSurvived >= check.value
  nationAtRelation      Any nation's relation is at or beyond check.value
                        in the order: indifferent → neutral → wary → suspicious
                        → notice → concern → alarm → hostile → war
  nationThreatReduced   Any nation's threat is currently 0 (was hostile before)

To add a new type: add a case to evaluateMilestoneCheck() in progression.js.
This is the one place where adding a new content type does require a JS edit,
but it's a single switch case.

  GAME STATE (js/state.js → GS object)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GS is the single source of truth for all runtime state. It is what gets
serialised to localStorage on save. All fields:

  bronze               Total wealth in bronze. 100b = 1s, 100s = 1g.
  day                  Current in-game day (increments every DAY_DURATION secs)
  commander            { name, origin, portraitSymbol, startingItem,
                         level, xp, xpNext, hp, maxHp, str, end, lead, cun }
  roster               Array of hired member objects (generated by recruits.js)
  activeJobs           Array of { jobId, timeLeft, totalTime }
  fame                 Numeric fame value
  fameRank             Index into DATA.fameRanks
  pmcTier              0-6, current PMC tier index
  upgrades             { upgradeId: true } for all built fort upgrades
  milestones           { milestoneId: true } for all achieved milestones
  totalJobsDone        Cumulative job count
  completedJobs        { jobId: count } per-job completion counts
  completedCategories  { category: count } per-category completion counts
  recruitPool          Currently offered recruit objects
  nationThreat         { nationId: numericThreat }
  nationRelations      { nationId: relationString }
  suppressionEvents    Array of active suppression ops:
                         { eventId, nationId, timeLeft, totalTime,
                           resolved?, accepted? }
  suppressionSurvived  Count of suppression events resolved
  suppressionPenalties { key: { type, value, categoryFilter?, expiresDay } }
  lastSuppressionCheck Day when suppression spawning was last evaluated
  territoryClaimed     Count of territorial contracts completed

When loading a saved game, new fields added since the save was created default
to safe values. See the Object.assign spread in state.js loadGame().

Every contract entry:

  id                   Unique string. Referenced by milestones, forbiddenIfActiveJobs.
  name                 Display name.
  desc                 One or two sentence description.
  flavour              Short italic quote shown under description.
  category             Controls grouping, icons, and category-filtered effects.
                       Current categories: scouting, escort, guard, bounty,
                       combat, siege, dungeon, arcane, intelligence, political,

  danger               1-4. Affects default fame if fameBase not set.

  rewardBase           Bronze reward (modified by engine at runtime).

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

NATIONS (nations.json):

  id                   String key used in nationThreatGain, GS.nationThreat, etc.
  type                 duchy | kingdom | theocracy | alliance | empire
                       Used to filter which suppression events can target them.
  toleranceThresholds  { notice, concern, alarm, hostile, war } — numeric threat
                       values at which each escalation level activates.
  responses            { notice, concern, ... } — flavour text shown in log
                       when that level is first reached.
  suppressionStrength  1-9. Informational — can be used for future mechanics.

  pmcTierNotice        Minimum PMC tier before this nation tracks you at all.
  triggersOnArcaneJobs Boolean (optional). Extra threat for arcane contracts.

ESCALATION ORDER:
  indifferent → neutral → wary → suspicious → notice → concern → alarm →
  hostile → war

Threat naturally decays by nationThreatDecayPerDay (config) every in-game day.
Intelligence Network and Diplomatic Corps upgrades reduce threat accumulation
multiplicatively. Shadow Operative recruit also reduces it.

SUPPRESSION EVENTS (suppression_events.json):

Nations spawn suppression events when at "alarm" relation or above, checked
every suppressionCheckIntervalDays (config). Events appear in the Contracts tab
as special red cards. The player has the duration window to click "Resolve Now".
If the timer expires without resolution, penaltyIfIgnored is applied.

Each event has:
  triggersAtRelation   Minimum relation before this event can spawn.
  sourceNationTypes    Which nation types can spawn it (or "all").
  effect               { type: "threatReduction", nation: "source"|id, value }
                       Applied on resolution.
  penaltyIfIgnored     { type, value, categoryFilter?, duration } applied if
                       the event expires unresolved.
  flavourResolved      Log message shown on successful resolution.

PMC TIERS (pmc_tiers.json):

  tier                 0-6 integer.
  thresholdFame/Bronze/Roster/JobsDone  All four must be met simultaneously.
  nationThreatMult     Multiplier applied to all nationThreatGain values while
                       at this tier. 0.0 = invisible, 5.0 = very threatening.
  unlocks              Array of job categories that become visible at this tier.

  HOW TO ADD NEW CONTENT (quick reference)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ADD A CONTRACT:
  Edit data/jobs.json. Copy an existing entry, change id/name/desc/requirements.
  Use an existing category or add a new string — if it's a PMC-tier category,
  add it to the appropriate tier's unlocks[] in pmc_tiers.json too.
  Add a category icon to game_config.json pmcTierCategoryIcons if needed.
  No JS changes required.

ADD A RECRUIT CLASS:
  Edit data/recruit_classes.json. Add an entry with bonuses[] using existing
  effect types. Set rarityWeight (higher = more common).
  No JS changes required.

ADD A FORT UPGRADE:
  Edit data/fort_upgrades.json. Add effects[] using existing effect types.
  If you need a new effect type, add a function to engine.js.
  No JS changes required for existing types.

ADD A MILESTONE:
  Edit data/milestones.json. Use an existing check.type.
  If you need a new check type, add one case to evaluateMilestoneCheck() in
  js/logic/progression.js. That's the only JS change needed.

ADD A NATION:
  Edit data/nations.json. Use an existing nation as a template.
  Add nationThreatGain entries pointing to the new nation ID in relevant jobs.
  No JS changes required.

ADD A SUPPRESSION EVENT:
  Edit data/suppression_events.json. Use existing penaltyIfIgnored types.
  The spawning, ticking, and resolution logic in nations.js handles all events
  generically — no JS changes required.

ADD A NEW GAME CONFIG VALUE:
  Edit data/game_config.json. Access it in JS as DATA.config?.yourKey ?? default.

CHANGE THE LAYOUT:
  Edit js/loader/dom.js. All game markup is here.

CHANGE THEMING:
  Edit css/style.css :root variables.

  TECH NOTES FOR AI AGENTS SPECIFICALLY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Before making changes, read the relevant data files first. The JSON is the
  specification. The JS is the interpreter. Changes to mechanics belong in data.

- Before adding a JS function, search engine.js. The accumulator pattern means
  many mechanics are already supported — you just need the right JSON entry.

- GS is mutable global state. DATA is immutable after load. Never write to DATA.

- All render functions are idempotent — calling renderAll() twice is safe.
  The game loop calls renderCurrency() and renderCommander() every frame.
  renderAll() is called when something significant changes.

- The log (log() in utils.js) accepts HTML strings. Use <em> and <strong>.
  Log types: 'info', 'good', 'bad', 'gold'. Gold is for level-ups/milestones.

- Saves are keyed by player-chosen name in localStorage with prefix
  "ironcompany_save_". The index list is at "ironcompany_saves". There is no
  save slot limit — the array grows unboundedly.

- The game must run over HTTP. file:// will silently fail on fetch() calls and
  the boot screen will show a data load error. Always use server_manager.pyw
  or python -m http.server.

- requestAnimationFrame loop runs continuously. dt (delta time in seconds) is
  calculated each frame. All timers in GS.activeJobs and GS.suppressionEvents
  decrement by dt, not by frame count.

- The engine does not yet handle partial effects during gameplay — if an upgrade
  is built while a job is in progress, the new effect applies immediately on
  the next getJobDuration/getJobReward call. This is intentional and fine.

  ARCHITECTURE PRINCIPLE:
  The existing codebase had a three-layer inconsistency: dom.js hardcoded the HTML
  skeleton, render/*.js hardcoded what went into the slots, and screens could only
  be restructured by editing JS. Session 17 collapses this into two clean layers:
    data/ui/screens.json  — what the UI looks like (layout, tabs, widgets)
    js/widgets/*.js       — how each widget type renders itself
  Adding a new panel, tab, or widget placement is now a JSON edit. Adding a new
  widget type requires one new JS file and one WidgetRegistry.register() call.

  NEW DATA FILE: data/ui/screens.json
  Fully declarative screen layout. Describes the header widget slots, overlay
  placements (notifications, char creation, save-select), and the two main panels
  (left + center). The center panel is described as a tab tree: tabs → sub-tabs →
  sub-sub-tabs, each leaf listing its widgets. Every widget entry has a "widget"
  type name (looked up in WidgetRegistry) and a "config" object passed to the
  constructor. Loaded by data.js in the same Promise.all as all other game data;
  accessible as DATA.screens after loadAllData().

  NEW FILE: js/widgets/Widget.js
  Base class for all widgets. Contract:
    mount()        — called once; builds initial DOM into this.container
    render()       — idempotent state sync; called by ScreenEngine.renderAll()
    tick(dt)       — per-frame update (only override if needed)
    destroy()      — cleanup listeners and timers
  Helper methods: el(tag, cls, html), setHTML(html), byId(id), emit(name, detail).
  All widgets are globals (no module system, consistent with the rest of the codebase).

  NEW FILE: js/ui/WidgetRegistry.js
  Maps widget type name strings to classes. register(name, klass) called at the
  bottom of each widget file. get(name) used by ScreenEngine. list() for debug.

  NEW FILE: js/ui/ScreenEngine.js
  Reads DATA.screens and builds the complete DOM into #app:
    _buildHeader()         → header element + overlay placeholder
    _buildOverlay()        → per overlay in screens.overlays[]

    _buildTabBar()         → main tab bar + tab panes; recurses into sub-tabs
    _buildSubTabBar()      → sub-tab bar + sub-tab panes; recurses into sub-sub-tabs
    _buildSubSubTabBar()   → sub-sub-tab bar + panes (used by Contracts > Active)

  REPEAT QUEUE BUG FIXES (js/main.js):

  Bug 1 — Auto-restart silently failing:
    Root cause: checkJobRequirements was called inside the GS.activeJobs.filter()
    callback while the completing job was still present in GS.activeJobs. The
    "already underway" check (engine.js line 267) fired, rechk.ok was false, and
    the job was removed from GS.repeatQueue rather than restarted.
    Fix: Collect jobs for restart in a _repeatRestarts array inside the filter
    callback. After filter() completes — when the job is fully removed from
    GS.activeJobs — iterate _repeatRestarts and run checkJobRequirements cleanly.
    Set completedAny = true after restart so renderAll fires immediately.

  Bug 2 — Must click manually to start a queued job:
    Root cause: addToRepeatQueue() only pushed the jobId to GS.repeatQueue with
    no logic to start an idle queued job immediately.
    Fix: After pushing to the queue, check whether the job is already in
    GS.activeJobs. If not, and if checkJobRequirements passes, push it onto
    GS.activeJobs immediately. The 🔁 button now means "add to repeat queue AND
    start now if not already running."
